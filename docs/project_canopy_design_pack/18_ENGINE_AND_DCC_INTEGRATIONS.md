# Engine and DCC integrations

## Objective

Project Canopy must move assets into production tools without requiring a proprietary interchange format. Integrations combine open-format import, Canopy metadata sidecars, target shaders, material conversion, LOD setup, wind hookup, validation and round-trip automation.

The common integration code is generated from the export manifest and the public runtime schema. Engine or DCC packages must not parse authoring project files directly.

## Common package architecture

Each integration contains:

- An importer or add-on
- A material and texture mapper
- Wind and animation support where the host permits it
- LOD and impostor setup
- Metadata and collision import
- A validator with actionable diagnostics
- Example assets generated by the project, not third-party commercial content
- Automated host-version smoke tests
- A compatibility matrix

The import manifest records tool version, source document hash, export preset hash, schema versions, coordinate convention, texture color spaces, material assignments, wind profile, LOD thresholds and generated files.

## Unity package

Package identifier: `org.project-canopy.importer` after final naming review.

### Import path

- Primary: glTF or a Canopy runtime asset through a custom `ScriptedImporter`
- Optional: FBX when the separately licensed exporter is present
- Textures imported with explicit sRGB or linear settings
- Generated materials for Built-in Render Pipeline, URP and HDRP
- Prefab creation with `LODGroup`
- Impostor terminal renderer
- Collision and named anchors
- Source manifest retained as a subasset

### Runtime behavior

- Wind domain bridge from scene wind components or a Canopy wind controller
- Material property blocks for per-instance tint, phase, season and growth age
- GPU instancing and indirect-draw option
- Terrain placement adapter
- Light probe, reflection probe and shadow settings
- Optional Entities Graphics integration in a separate package

### Validation

The importer reports missing maps, wrong color spaces, incompatible shaders, absent tangents, unsupported packing recipes, inconsistent LOD bounds and unavailable render-pipeline features.

### Unity acceptance scene

The package includes a scene with:

- Geometry-to-impostor LOD transition
- Three wind profiles
- Seasonal material blend
- One million-instance stress mode using indirect rendering
- URP and HDRP reference captures
- Mobile lightweight-wind profile

## Unreal Engine plugin

Module layout:

- `CanopyImporter`: asset factory, reimport and metadata
- `CanopyRuntime`: optional native runtime asset and forest components
- `CanopyEditor`: UI, validation and placement tools
- `CanopyShaders`: reusable material functions and optional global shaders

### Import path

- Primary: glTF, USD or `.canopyrt`
- Static Mesh assets per discrete LOD where host-native assets are desired
- Material instances derived from versioned master materials
- Texture virtualisation and compression settings from semantic metadata
- Collision primitives, sockets and pivot information
- Reimport keyed by source and preset hashes

### Rendering policy

- Conventional LODs and impostors are normative.
- Nanite use is an optional target profile, selected only where current engine constraints and project requirements permit it.
- Wind data maps to documented vertex semantics and shared material functions.
- Foliage placement can use native foliage actors or the Canopy population component.
- Shadow, virtual shadow map and ray-tracing policies are preset controlled.

### Unreal acceptance map

The sample demonstrates masked two-sided leaves, bark/frond materials, wind, color variation, world-position precision, LOD fading, impostors, foliage placement, collision, metadata, reimport and packaged-build operation.

## Blender add-on

The Blender add-on targets supported LTS and current stable releases.

Capabilities:

- Import glTF, USD or Alembic with Canopy sidecar metadata
- Build material node graphs for bark, leaves, fronds and impostors
- Create collections per LOD and geometry class
- Import point-cache, shape-key or armature animation as appropriate
- Preserve custom vertex attributes
- Create Geometry Nodes instance populations from Canopy placement data
- Export edited geometry back through standard formats without pretending to round-trip the procedural graph
- Batch import through background Blender

Wind preview can use geometry nodes or shader deformation; baked VFX output remains the interchange authority.

## DCC interchange rules

- USD is preferred for scene structure, point instancing, variants and MaterialX.
- Alembic is preferred for topology-varying or dense point-cache animation.
- glTF is preferred for portable real-time static assets and simple animation.
- OBJ is diagnostic and static only.
- FBX is an optional compatibility path, not a canonical internal dependency.
- MaterialX is preferred where the target supports it; otherwise use a documented native conversion.

## Coordinate and units contract

Every integration declares:

- Source and target handedness
- Up and forward axes
- Linear unit conversion
- UV origin and tangent convention
- Normal-map convention
- Transform bake policy
- Animation frame rate and time basis

Automated asymmetric test assets verify translation, rotation, scale, normals, tangents, winding, sockets and animation.

## Shader package

The shared shader specification defines functions for:

- Hierarchical branch wind
- Leaf ripple and tumble
- Frond motion
- Crossfade and dithering
- Two-sided leaf normal handling
- Subsurface/transmission approximation
- Per-instance hue, season and variation
- Impostor view selection and depth reconstruction
- Motion vectors under wind
- Alpha-to-coverage policy

Each engine implementation includes conformance images generated from the same parameter vectors.

## Version policy

- Integrations support explicit host-version ranges.
- `main` follows current supported host versions; maintenance branches cover older hosts.
- Unsupported host APIs are isolated behind adapters.
- Package versions follow semantic versioning independently from the core product.
- An integration release states the exact core and schema versions tested.

## Integration release gate

For each supported host:

- Clean installation succeeds.
- Import and reimport succeed in GUI and batch mode where available.
- Materials, units, axes, LOD, wind, animation, collision and metadata match the reference manifest.
- Project packaging or rendering succeeds without editor-only dependencies.
- Automated sample captures meet perceptual thresholds.
- Uninstall removes generated package code without deleting user assets.
